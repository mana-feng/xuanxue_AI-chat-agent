# 密码存储安全说明

## ⚠️ 重要安全警告

### 为什么不建议前端发送密码 Hash 给后端？

**如果前端存储密码 hash 并发送给后端对比，存在严重安全隐患：**

1. **Hash 等同于密码**：如果攻击者获取到 hash，可以直接用 hash 登录，无需原始密码
2. **违背密码学原则**：密码 hash 应该只在服务器端使用，不应该在网络传输
3. **无法防止重放攻击**：如果 hash 泄露，攻击者可以无限次使用该 hash 登录

### 正确的做法

1. **前端存储密码 hash**：仅用于"记住密码"功能的本地验证（用户输入密码后，对比 hash 确认是否匹配）
2. **登录时发送原始密码**：用户输入密码后，发送明文密码给后端（HTTPS 加密传输）
3. **后端验证**：后端使用 bcrypt 对比数据库中的 hash

## 当前实现方案

### 1. 密码 Hash 存储（前端）

- **用途**：仅用于"记住密码"功能
- **存储方式**：AES 加密存储密码的 SHA-256 hash
- **位置**：`src/utils/passwordHash.ts`
- **方法**：
  - `hashPassword(password)` - 生成密码 hash
  - `verifyPasswordHash(password, storedHash)` - 本地验证密码

### 2. 登录流程

```
用户输入密码
    ↓
前端生成 hash（用于记住密码）
    ↓
发送原始密码给后端（HTTPS）
    ↓
后端 bcrypt 验证
    ↓
返回 JWT token
```

### 3. 记住密码功能

- 用户可以选择"记住密码"
- 如果选择，存储密码的 hash（加密）
- 下次打开登录页时，提示用户是否使用记住的密码
- **但登录时仍然需要用户输入密码**（或自动填充后用户确认）

## 安全建议

1. **生产环境必须使用 HTTPS**：确保密码传输安全
2. **不要在前端存储原始密码**：只存储 hash
3. **不要发送 hash 给后端**：始终发送原始密码
4. **定期更新密码**：建议用户定期更换密码
5. **使用强密码策略**：后端已有密码验证

## 如果必须实现自动登录

如果确实需要自动登录功能，建议：

1. **使用 Refresh Token**：
   - Access Token 短期有效（如 1 小时）
   - Refresh Token 长期有效（如 30 天）
   - 前端存储 Refresh Token（加密）
   - 使用 Refresh Token 获取新的 Access Token

2. **生物识别**：
   - 使用设备指纹或生物识别（指纹、Face ID）
   - 结合 token 机制实现自动登录

3. **设备信任**：
   - 首次登录后标记设备为"信任设备"
   - 信任设备可以使用长期 token 自动登录
